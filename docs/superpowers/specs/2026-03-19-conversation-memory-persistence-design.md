# 会话记忆、历史持久化与鉴权收口设计

## 1. 背景

当前系统的聊天链路存在三个核心问题：

1. 前端本地状态是事实来源，刷新或切换后无法可靠恢复。
2. 后端 `/chat` 只接收单条 `prompt`，没有基于历史会话继续对话的能力。
3. 认证已接入 `better-auth`，但聊天会话、历史读取、权限边界还没有真正和用户体系打通。

这直接导致：

- 用户发完第一句，再发第二句时，模型可能拿不到上文。
- 历史对话无法可靠持久化，也不能从左侧会话面板恢复。
- 已登录和未登录状态下的聊天边界不够清晰，后端会话接口还没有完全收口到用户维度。

## 2. 目标

本次改造的目标如下：

- 让同一历史会话中的后续消息基于完整消息历史继续对话。
- 把每个已登录用户的会话、消息、tool 调用、task 状态持久化存储。
- 让前端左侧会话面板渲染当前登录用户自己的历史会话列表。
- 让用户可以重新打开任一历史会话并继续对话。
- 保持 tool 调用和 task 状态在消息流中附属展示，而不是拆到独立面板。
- 收紧 `better-auth` 鉴权边界，让所有会话相关接口都以服务端 session 为准。
- 把测试纳入首期交付，而不是作为后补项。

## 3. 非目标

本次不包含以下内容：

- 邮箱验证、找回密码、多端设备管理等更完整的账号能力扩展。
- 事件溯源式的全量运行日志回放系统。
- 跨用户共享会话。
- 游客长期持久化历史会话。

未登录用户只保留临时会话态，不进入正式历史持久化体系。

## 4. 设计原则

- 代码边界清晰，避免把运行时逻辑、持久化逻辑、鉴权逻辑混在一起。
- 消息上下文和运行观测分层存储，避免模型上下文被 tool/task 噪音污染。
- 服务端是事实来源，前端不再自建临时 `threadId` 作为长期标识。
- 接口设计优先匹配当前 UI 交互，不额外引入复杂概念。
- 所有核心链路都要有可执行测试覆盖。

## 5. 架构概览

### 5.1 分层方案

采用“运行时会话”和“持久化会话”分层方案：

- `AgentSession`
  - 只负责单次 agent 运行。
  - 处理消息推进、SSE 事件、tool 调用和 task 事件发射。
- `ConversationService`
  - 负责会话创建、会话归属校验、历史装载、消息持久化、tool/task 持久化、会话列表查询。
- `better-auth`
  - 负责用户认证和服务端 session 获取。
  - 所有会话接口都以服务端 `user.id` 为唯一隔离键。
- 前端会话工作区
  - 负责列表渲染、历史详情渲染、消息发送和 SSE 更新。
  - 不负责生成长期会话标识，也不负责决定哪些历史进入模型上下文。

### 5.2 核心判断

用户继续对话时：

1. 前端只传 `conversationId + prompt`。
2. 服务端校验会话归属当前用户。
3. 服务端读取完整历史消息。
4. 若消息过长，由服务端做压缩或裁剪。
5. 服务端把压缩后的历史和本轮用户消息注入新的 `AgentSession`。
6. 运行结束后，把本轮消息和观测数据持久化。

## 6. 数据模型

### 6.1 conversations

用途：左侧会话列表、归属控制、会话元信息。

建议字段：

- `id`
- `userId`
- `title`
- `status`
- `summary`
- `messageCount`
- `lastMessageAt`
- `createdAt`
- `updatedAt`

说明：

- `userId` 是所有权限校验的上游锚点。
- `summary` 预留给长上下文压缩，不要求首期就做复杂摘要策略。
- `title` 首期可采用首条用户消息截断生成，后续可支持用户重命名。

### 6.2 conversation_messages

用途：会话正文，既用于历史展示，也用于模型续聊上下文。

建议字段：

- `id`
- `conversationId`
- `role`
- `content`
- `sequence`
- `status`
- `createdAt`
- `tokenCount`

说明：

- `role` 首期支持 `system | user | assistant`，必要时允许 `tool` 扩展。
- `sequence` 负责全局排序，保证历史恢复顺序稳定。
- `status` 至少支持 `streaming | complete | failed | partial`。

### 6.3 conversation_tool_calls

用途：消息流中的 tool 附属展示，不直接参与模型上下文。

建议字段：

- `id`
- `conversationId`
- `messageId`
- `name`
- `argsJson`
- `resultText`
- `status`
- `sequence`
- `startedAt`
- `completedAt`

说明：

- `messageId` 关联到所属 assistant 消息轮次。
- `sequence` 用于恢复和消息流一致的可视顺序。

### 6.4 conversation_task_events

用途：消息流中的 task 附属展示，不直接参与模型上下文。

建议字段：

- `id`
- `conversationId`
- `taskId`
- `subject`
- `status`
- `sequence`
- `updatedAt`

说明：

- 这里存的是“对 UI 有价值的任务状态变化”，不是所有内部对象快照。
- 前端渲染时可聚合为当前 task 状态，也可按事件序列展示。

## 7. 接口设计

### 7.1 保留接口

- `GET /api/me`
  - 作用：前端初始化登录态。

### 7.2 新增接口

- `GET /api/conversations`
  - 返回当前用户的会话列表。
- `POST /api/conversations`
  - 创建空会话，或创建带初始标题的会话。
- `GET /api/conversations/:id`
  - 返回某个会话的完整详情，包括消息流渲染所需聚合数据。
- `POST /api/conversations/:id/messages`
  - 在指定会话内继续对话，并以 SSE 返回运行过程。
- `PATCH /api/conversations/:id`
  - 首期可只支持重命名和归档。

### 7.3 现有 `/chat` 的处理

现有 `POST /chat` 不再继续作为正式前端聊天入口。

建议：

- 对外收口到基于 conversation 的受保护接口。
- 若短期保留 `/chat`，也应仅作为内部过渡实现，不再被前端直接依赖。

## 8. 后端运行流程

### 8.1 创建会话

1. 前端确认用户已登录。
2. 前端请求 `POST /api/conversations`。
3. 服务端根据当前 session 的 `user.id` 创建会话。
4. 返回 `conversationId` 给前端。

可选策略：

- 显式创建空会话。
- 首条消息发送时懒创建。

首期更推荐显式创建空会话，状态与左侧 panel 更一致，也更容易测试。

### 8.2 继续对话

1. 前端调用 `POST /api/conversations/:id/messages`，body 为 `prompt`。
2. 服务端通过 `better-auth` 获取当前用户。
3. 服务端校验该会话归属当前用户。
4. 服务端读取该会话全部历史消息。
5. 若历史过长，调用上下文压缩逻辑，产出可继续运行的消息集。
6. 服务端先持久化本轮 user message。
7. 服务端创建一次新的 `AgentSession`。
8. 运行过程中把 SSE 事件推给前端，同时写入 tool/task 观测数据。
9. 结束后持久化 assistant message，并更新会话元信息。

### 8.3 历史恢复

1. 前端打开某会话详情页或切换会话。
2. 请求 `GET /api/conversations/:id`。
3. 服务端返回消息、tool、task 聚合后的渲染数据。
4. 前端据此恢复完整消息流显示。

## 9. 前端设计

### 9.1 左侧会话面板

左侧 panel 负责：

- 加载当前登录用户的会话列表。
- 显示标题、更新时间、必要状态。
- 支持创建新会话。
- 支持切换到历史会话。

数据来源必须是服务端 `GET /api/conversations`，不再依赖前端本地线程数组。

### 9.2 主聊天区

主聊天区负责：

- 渲染当前会话消息。
- 在 assistant 消息下附属展示对应的 tool 调用。
- 在消息流中附属展示 task 状态。
- 在发送消息时针对当前 `conversationId` 继续对话。

### 9.3 本地状态职责

前端本地状态仍然需要保留，但职责收缩为：

- 保存当前打开的 `conversationId`。
- 保存当前详情页的渲染态。
- 合并本次 SSE 增量，提升交互实时性。

前端本地状态不再承担长期事实来源职责。

## 10. 鉴权设计

### 10.1 服务端边界

所有会话相关接口必须统一：

- 先通过 `auth.api.getSession` 获取服务端 session。
- 只信任 session 中的 `user.id`。
- 查询和写入都附带 `userId` 归属约束。

禁止：

- 前端传 `userId` 参与权限判断。
- 通过前端缓存决定当前用户是否有权访问某个会话。

### 10.2 前端登录态

前端要明确三种状态：

- `loading`
- `unauthenticated`
- `authenticated`

要求：

- 未登录时不加载会话列表、不进入正式聊天工作区。
- 登录后才初始化会话相关数据。
- 登录失效时清空会话本地态，并回到认证壳层。

## 11. 错误处理

### 11.1 权限类错误

- 未登录：返回 `401`。
- 会话不属于当前用户：建议统一返回 `404`，减少资源探测面。

### 11.2 运行类错误

- 如果 agent 运行失败，保留本轮 user message。
- assistant 侧新增失败态记录，或把本轮 assistant message 标记为 `failed/partial`。
- SSE 中途断开时，已持久化数据保留，不回滚。

### 11.3 数据完整性

- tool 调用允许存在 `running` 未完成状态。
- task 事件允许只有部分更新。
- 详情恢复时以前端可解释为准，不强求每条观测数据都形成完美闭环。

## 12. 测试策略

### 12.1 单元测试

覆盖以下核心逻辑：

- `ConversationService` 的归属校验。
- 历史消息装载与排序。
- tool/task 到消息流附属展示的聚合映射。
- 长上下文压缩触发条件。

### 12.2 集成测试

覆盖以下主链路：

- 已登录用户创建会话。
- 同一会话连续两轮对话，上下文可延续。
- 重新打开历史会话后，消息、tool、task 可恢复。
- 用户 A 不能读取用户 B 的会话。
- SSE 过程中消息和观测数据能够持久化。

### 12.3 前端测试

覆盖以下场景：

- 未登录时不会进入聊天主流程。
- 登录后正确拉取会话列表。
- 左侧 panel 切换会话后正确恢复详情。
- 当前会话继续发消息时，SSE 增量和历史数据能正确合并。

## 13. 验收标准

- 同一历史会话中，第二句能够基于第一句上下文继续对话。
- 刷新页面后，历史会话依然能完整恢复。
- 左侧会话面板展示当前登录用户自己的会话列表。
- 历史详情中能看到消息、tool 调用、task 状态。
- 跨用户访问任一会话接口都会被拒绝。
- 至少存在一条后端集成测试覆盖“创建会话 -> 连续对话 -> 再次打开历史”的主链路。

## 14. 分阶段实施建议

### 阶段一

- 建立会话与消息表。
- 打通 conversation API。
- 前端左侧 panel 接入会话列表。
- 消息历史可恢复。

### 阶段二

- 持久化 tool 调用和 task 状态。
- 历史详情中恢复附属展示。

### 阶段三

- 接入上下文压缩和会话摘要。
- 增强会话管理能力，例如重命名、归档。

该分阶段仅用于实施顺序，不改变首期设计边界。首期交付仍需覆盖完整主链路与必要测试。
